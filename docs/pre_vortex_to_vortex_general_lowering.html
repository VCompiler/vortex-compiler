Pre-Vortex 到 Vortex 的通用 Lowering 设计

这份页面强调的是“通用 lowering 规则”，而不是只针对某一个 matmul 样例做逐行替换。 核心目标是：只把执行、内存、同步这些真正的硬件语义 materialize 成 vortex.*。

func arith scf memref tensor linalg vector vortex

推荐目标不是“纯 Vortex IR”，而是 混合 IR：
func + arith + scf + memref + vector/linalg + vortex

1. 通用 Lowering 总流程

1

规范化 Pre-Vortex

  • canonicalize / CSE
  • loop normalization
  • tiling / fusion
  • tensor → memref bufferization
2

Mapping Analysis

  • 识别 kernel 入口
  • 识别并行维 / reduction 维
  • 决定 core / subgroup / thread 映射
  • 选出 local promotion 候选
3

Materialize Vortex 语义

  • vortex.kernel
  • vortex.launch
  • core_id / subgroup_id / thread_id
  • local_alloc / barrier / fence
4

Vortex-aware Low Level IR

  • 继续 lower 剩余 linalg/vector
  • 保留标准 scf/memref/arith
  • 为 LLVM lowering 做准备

2. 设计原则

原则 A：只下沉硬件语义

执行层级、地址空间、local memory、同步语义进入 Vortex；纯算法与普通控制流尽量保留标准 MLIR。

原则 B：不是所有 op 都替换

arith、大部分 scf、大部分 memref、大部分 linalg 都不应该机械重命名为 vortex.*。

原则 C：结果应更接近 LLVM

如果换成 vortex.* 只是“改名”，但不能更稳定地进入 LLVM backend，就不应引入。

3. 当前 Vortex Dialect 能力边界

已能表达

  • kernel 边界：vortex.kernel
  • 执行区域：vortex.launch
  • 执行 id：core_id / subgroup_id / thread_id
  • 地址空间：#vortex.address_space<...>
  • local memory：vortex.local_alloc
  • 同步：vortex.barrier / vortex.fence

暂时缺失

  • 完整 grid/block 级 launch 语义
  • 全局 tile 网格 id
  • 矩阵专用原语（如 vortex.mma）
  • 显式 predication / mask
  • local bank/layout 细节

所以当前更适合生成“混合 Vortex IR”，而不是试图把所有并行结构都强行吸收到当前 dialect。

4. Op 类别决策表

类别 典型 op 通用 lowering 策略 是否直接进 Vortex
func func.func, func.return 保留 func.func，对 kernel 加 vortex.kernel，参数补 global address space 部分进入
arith constant, addi, mulf 继续保留标准 SSA 算术 否
scf for, if, while 算法循环保留；映射到执行层级的区域才重组为 vortex.launch 选择性
memref subview, load/store, alloc subview 多数保留；真正 promotion 到 local 时改成 vortex.local_alloc 选择性
linalg generic, fill, matmul 先 tile/bufferize/vectorize；当前不建议直接变成 vortex.matmul 通常否
vector transfer_*, contract 保留为中间层；未来有硬件原语再识别成专用 Vortex op 通常否
tensor extract_slice, insert_slice 先 bufferization，再进入 Vortex lowering 否

5. 四种核心 Lowering 模式

Kernel 边界 materialization

func.func @kernel(%a: memref<...>)
↓
func.func @kernel(
  %a: memref<..., #vortex.address_space<global>>)
  attributes {vortex.kernel}

执行映射 materialization

... mapped tile region ...
↓
vortex.launch %cores, %subgroups, %threads {
  %c = vortex.core_id : index
  %s = vortex.subgroup_id : index
  %t = vortex.thread_id : index
}

Local promotion

memref.subview(global tile)
↓
vortex.local_alloc() : memref<..., #vortex.address_space<local>>
cooperative copy
vortex.barrier <core>

Structured compute 保留/下沉

linalg.matmul / linalg.generic
↓
可暂时保留在 launch 内
或继续 lower 为
scf + memref + arith

6. 不同 Kernel 类型的通用路线

Tiled Matmul

  • outer tile loops 可继续保留 scf.for
  • inner cooperative tile region → vortex.launch
  • A/B/C tile promotion → vortex.local_alloc
  • copy 后插 vortex.barrier <core>
  • 计算体可暂时保留 linalg.matmul 或继续 lower

Elementwise / Map / Reduction

  • elementwise：按并行维映射到 subgroup/thread
  • reduction：先保留结构，再决定 subgroup/core 规约策略
  • 当前阶段不必先造 vortex.reduce
  • 优先用标准 scf + arith + memref 承载计算本体

7. 推荐 Pass Pipeline

A

Pre-Vortex Normalize

  • canonicalize / CSE
  • bufferization
  • tiling / fusion
B

Analysis

  • parallel/reduction 维识别
  • launch mapping
  • promotion candidate 识别
C

Materialize Vortex

  • vortex.kernel
  • vortex.launch
  • local_alloc
  • barrier/fence
D

Prepare LLVM

  • lower 剩余 linalg/vector
  • 压低到 scf/memref/arith
  • 进入 VortexToLLVM

8. 当前阶段最重要的现实判断

不要把“某个样例 IR 里出现了什么 op”直接等同于“Vortex dialect 需要有一个同名替代 op”。

更通用的理解应该是：

  • 标准 MLIR 继续描述算法和结构
  • Vortex 只负责表达已经确定的硬件边界
  • 真正需要优先做的是 语义型 lowering pass，而不是算子重命名
